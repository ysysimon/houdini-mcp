# Roadmap: Workflow Instructions in FastMCP

## Goal

Add a routing layer to the FastMCP `instructions` string that directs Claude to read relevant workflow docs before attempting Houdini tasks. The workflow docs are auto-generated from the user's Houdini install (via the .hip ingest pipeline) and then refined with a manual pass.

The pattern: **instructions tell Claude *where to look*, workflow docs tell Claude *what to do*.**

## Design Principles

- **Generated first, then refined.** `ingest_hips.py` auto-generates workflow docs from `.hip` files. User does a manual pass to extend and correct. Committed to repo after refinement.
- **Always reference hip_patterns/.** The FastMCP instructions always mention `hip_patterns/` regardless of whether ingest has run. If the directory doesn't exist, Claude skips it gracefully — no conditional logic needed.
- **Routing is cheap.** The FastMCP instructions just point to files. The files contain the actual knowledge. Instructions stay short.

## Architecture

```
FastMCP instructions (always visible to Claude)
  ├── Existing: connection rules (1-8)
  └── New: workflow routing rules
        "Before creating geometry → search_docs or read docs/workflows/geometry.md"
        "Before setting up lighting → search_docs or read docs/workflows/lighting.md"
        "Before rendering with Mantra → search_docs or read docs/workflows/mantra.md"
        "Also search hip_patterns/ for version-specific examples"

docs/workflows/                     ← committed, auto-generated + manually refined
  ├── geometry.md
  ├── lighting.md
  └── mantra.md

hip_patterns/                       ← gitignored, generated per-machine
  ├── full_scenes/                  ← complete scene graphs
  ├── sop_networks/                 ← focused SOP subgraphs
  ├── rop_configs/                  ← focused render configurations
  ├── lighting_rigs/                ← focused lighting setups
  └── ...
```

## Decisions

- **Curation model:** Auto-generated by ingest pipeline, then manually refined and committed. Not purely hand-written, not purely automated.
- **Cold-start handling:** FastMCP instructions always mention `hip_patterns/`. If it doesn't exist, Claude finds nothing there and falls back to `docs/workflows/`. No dynamic instruction logic.
- **Search tool:** `search_docs` is extended to cover both `houdini_docs/` and `hip_patterns/`. One tool, multiple indices.
- **LLM flexibility:** Annotation is driven via micro-MCP server (see ROADMAP_HIP_INGEST.md). Any MCP client can do it — no assumption about Claude Code, Claude Desktop, or any specific LLM. Workflow docs benefit from annotations but don't require them.

## Phase 1: Auto-Generated Workflow Docs — Core Three

**Goal:** The ingest pipeline (ROADMAP_HIP_INGEST.md) generates initial workflow docs from `.hip` files. User refines them manually.

### `docs/workflows/geometry.md`
- [ ] Auto-generated: common SOP patterns found in example `.hip` files
- [ ] Manual additions: OBJ → GEO node → SOPs workflow, naming conventions
- [ ] Procedural patterns: copy to points, scatter, boolean, extrude
- [ ] When to use `execute_houdini_code` vs. node creation tools
- [ ] Common mistakes Claude makes (wrong node context, missing display flag)

### `docs/workflows/lighting.md`
- [ ] Auto-generated: light node configurations found in example scenes
- [ ] Manual additions: GI/bounce setup, which parameters matter
- [ ] Light types (hlight, environment, sun, area), environment HDRI
- [ ] Three-point lighting as a baseline pattern
- [ ] Common mistakes (lights at origin, zero intensity, wrong light category)

### `docs/workflows/mantra.md`
- [ ] Auto-generated: Mantra ROP configurations from example scenes
- [ ] Manual additions: camera positioning relative to scene bounds, quality settings
- [ ] Material assignment workflow (principled shader → assign to geo)
- [ ] Common mistakes (no camera, no output path, missing materials = grey)
- [ ] When to use `monitor_render` for long renders

**Source material:**
- Auto-generated from `.hip` file ingest (primary source)
- Existing fetched Houdini docs (`search_docs` / `get_doc`)
- SideFX online documentation and tutorials
- User's manual refinement pass

**Output:** Three markdown files in `docs/workflows/`. Auto-generated, manually refined, committed to the repo. Written for Claude — concise, step-oriented, parameter-specific.

## Phase 2: FastMCP Instruction Routing

**Goal:** Add routing rules to the `instructions` string so Claude reads the right doc before acting.

- [ ] Add a "Workflow Reference" section to the FastMCP `instructions` in `houdini_mcp_server.py`
- [ ] Rules format:
  ```
  9. **Read workflow docs before complex tasks.** Before building a scene or network:
     - Geometry creation or SOP work → use search_docs("geometry workflow") or read docs/workflows/geometry.md
     - Lighting setup → use search_docs("lighting workflow") or read docs/workflows/lighting.md
     - Mantra rendering → use search_docs("mantra workflow") or read docs/workflows/mantra.md
     Also search hip_patterns/ for version-specific examples from the local Houdini install.
     Do NOT guess workflows — check the docs first.
  ```
- [ ] Always reference `hip_patterns/` — Claude skips gracefully if it doesn't exist
- [ ] Keep instructions concise — routing only, not the workflows themselves
- [ ] Test: verify Claude reads the docs when prompted with relevant tasks

**Output:** Updated `instructions` string in `houdini_mcp_server.py`.

## Phase 3: Integration with Generated Patterns

**Goal:** Connect the workflow docs to the full + focused patterns generated by the .hip ingest pipeline.

- [ ] `search_docs` covers both `houdini_docs/` and `hip_patterns/` transparently
- [ ] Workflow docs reference `hip_patterns/` subdirectories:
  - `geometry.md` → "search hip_patterns/sop_networks/ for SOP chain examples"
  - `mantra.md` → "search hip_patterns/rop_configs/ for Mantra configurations"
  - `lighting.md` → "search hip_patterns/lighting_rigs/ for lighting setups"
- [ ] Full scene graphs in `hip_patterns/full_scenes/` available for broader context
- [ ] Claude gets: refined workflow guide + version-specific real examples

**Output:** Workflow docs enriched with version-matched patterns from the user's Houdini install.

## Phase 4: Expand Workflow Coverage

**Goal:** Add workflow docs as new use cases arise. Each follows the same model: auto-generated from ingest, manually refined.

Candidates (rough priority):
- [ ] `docs/workflows/materials.md` — shader setup, texture assignment, principled shader params
- [ ] `docs/workflows/cameras.md` — camera types, positioning strategies, DOF, motion blur
- [ ] `docs/workflows/animation.md` — keyframing, expressions, CHOPs
- [ ] `docs/workflows/simulation.md` — DOPs, pyro, FLIP, RBD basics
- [ ] `docs/workflows/usd.md` — LOPs/Solaris scene assembly
- [ ] `docs/workflows/pdg.md` — TOPs pipeline patterns
- [ ] `docs/workflows/hda.md` — digital asset creation patterns

## Phase 5: Feedback Loop

**Goal:** Improve workflow docs based on observed Claude behavior.

- [ ] When Claude makes mistakes despite workflow docs, update with explicit "do NOT" guidance
- [ ] Track which docs get read most (bridge logging) to prioritize improvements
- [ ] User can flag bad behavior → triggers doc update
- [ ] Re-running the ingest script after a Houdini upgrade regenerates patterns; user does another manual refinement pass
